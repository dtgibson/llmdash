llmdash · multi-host badge · design draft

One glyph for every machine you watch — and which one is tight.

The badge grows from one machine to several. The glyph now shows the most-constrained window across all monitored machines and names which machine binds, alongside the existing C / X tool cue — ▪ Desktop·C 12%. The dropdown carries one honest section per host. Offline hosts are named; a monitoring station's empty local reading is de-emphasized but never faked. When you watch just one machine, it's byte-for-byte today's badge.

Draft · first direction · react and iterate

The host cue — form to ratify (OQ-05)

The glyph must name the binding host without getting busy. Below is the proposed form and the two alternatives considered. All stay xbar-safe (plain text + color=), and all collapse to today's badge exactly when only one machine is watched.

▪Desktop·C12%
host·tool ✓ proposed
Full short label, a tight · binding host to tool, then the number. Host and tool read as one token; the number still lands first. Recommended.
▪DT·C12%
initials
Shortest (DT·C), but a 2-letter initial is guessable, not legible — you re-learn which machine DT is. Saves space you don't need.
▪Studio VM…·X8%
truncation rule
A long label truncates at ~10 chars with an ellipsis (Studio VM…). The full name is in the dropdown header; the glyph stays one clean token.
Proposed rule: ▪ <host>·<C|X> <pct> in multi-host mode. Host label rendered at full length up to 10 characters, then truncated with … (the dropdown always carries the full escaped label). The · is a hairline separator, not a state marker — it never collides with the aging trailing · because the aging dot sits after the number, the host separator before the tool cue. In single-host mode the entire host cue is omitted — the glyph is ▪ C 12%, exactly today's badge.

The five honesty states — carried per binding host

The shipped five-state glyph grammar is unchanged; the host cue simply rides in front of the tool cue. The binding host's freshness band drives the glyph state, exactly as the binding tool's band does today. Offline (no server) is the badge's own local instance being down — distinct from a single remote host being unreachable, which is named per-host in the dropdown, never in the glyph.

The dropdown — one honest section per machine

Click the glyph and SwiftBar renders the plugin's dropdown: a title echo naming the binding machine, then one section per host — each a host header (escaped label + a freshness/offline pill) wrapping that host's Claude and Codex rows. Binding host leads; then reachable hosts in config order; the unreachable host named honestly; the empty local machine de-emphasized at the bottom, still shown.

How to read the sections, top to bottom:
  • Title echo — repeats the glyph and spells out the binding machine, tool, and window: Desktop · Claude · 5-hour. A Watching 4 machines · 1 not reachable line sets the scope honestly.
  • Desktop leads because it binds (a small binding pill marks it). Its rows are the exact per-tool rows the badge renders today — N% · resets …, and Claude 5-hour reads 12% in crit.
  • Work laptop follows in config order, fresh. Its Claude rows honestly read not available (it only runs Codex), never a fabricated 0%.
  • Studio VM is named unreachable — a per-host offline note carrying which machine and why, reusing the dashboard's stale-note grammar. Never a zero, never stale-as-fresh, never dropped.
  • This machine sits last, dimmed, labeled no local activity — retained honestly but out of the glance (the monitoring-station treatment).
The honesty floor holds per host. One host aging, stale, or offline never flags or suppresses another host's section — each carries its own freshness/offline state (the shipped five states, now per machine). Every host label is escaped before it touches a line; the Open dashboard link uses this machine's own loopback, never a peer's.

Editing the watched set — from the dropdown, no restart

The dropdown's bottom carries the live-config actions. Add host… opens a native macOS dialog; Remove host… opens a submenu listing the removable machines (never This machine); Watching: N hosts is a live listing. Each writes the local config file — no HTTP mutation — and the change applies on the next update.

The action set:
  • ＋ Add host… — opens the macOS input dialog (next section) to collect a host[:port][=label].
  • － Remove host… — a submenu, one item per removable host with its address. This machine is never listed (it's always present). Selecting one opens a confirm dialog.
  • ☰ Watching: 3 hosts — a live listing of the current remote set (escaped labels), a real affordance, not a dead item. Counts the 3 configured remotes; the title echo's "4 machines" includes this machine.
  • ◱ Open dashboard / ⟳ Refresh — the shipped actions, unchanged.
After any edit the badge refreshes so the dropdown reflects the new list, and the actual monitoring change takes effect on the next poller update — the copy never claims it's live before the poller has re-read the file. No plist edit, no service restart.

Add a host — the native dialog and its honest responses

Add host… runs a fixed-literal osascript display dialog on the same machine (no HTTP, no shell interpolation of the typed value). The entered string is sanitized, validated against the same grammar LLMDASH_HOSTS uses, deduped, and — only if valid — appended atomically. Malformed and duplicate entries are rejected with an honest message and nothing is written.

Reuse over reinvention. Every color, pill, note, and row grammar above is the established design system — status good/warn/crit tokens, the age-pill vocabulary (aging→warn, stale→crit), and the .stale-note callout — now applied per machine on the menu-bar surface.  ·  The only new chrome is the host header row and the host-level freshness/offline pill, mirroring the multi-host dashboard's .host-head / .host-pill treatment.  ·  xbar-safe floor: the host cue is plain text + color=, like every other glyph element — nothing depends on a SwiftBar-only param.  ·  Single-host is untouched: when one machine is watched, none of the host chrome renders — the badge is exactly what shipped.
