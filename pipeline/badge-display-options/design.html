llmdash · badge display options · design draft

Shape the glance: your machines, your layout, your width.

The badge stops insisting on one thing. A new Display submenu lets you choose which machines the glyph shows, whether to group by machine or by tool, how they're laid out (one, side-by-side, or rotating), and how dense they are (today's wide text, or a compact icon+number). A few named presets set it all at once; the axes stay adjustable underneath. All five honesty states survive the compact surface — a colored marker carries each, and no-reading / offline never show a fake number. With nothing set, the badge is byte-for-byte today's — ▪ Studio·C 12%.

Draft · first direction · react and iterate

The glyph, in every display mode — one real fleet

One scenario throughout: you watch Studio (Claude 5-hour at 12%, tight — the binder), Laptop (Codex at 88%, aging), and Desktop (Codex at 63%, fresh). The same three machines, rendered four ways. Density and layout change the shape of the glance; they never change the numbers or which machines are watched.

Compact · alternating — the menu bar shows one at a time, rotating every ~5s

The badge re-renders every 5 seconds; in alternating it rotates which single machine it shows each tick — one clean cell, its own color, its own state. The cue names which machine you're seeing right now, so it's never a guess. Deterministic (driven by the clock), so two ticks apart always advance to the next machine.

tick · +0s ▪St12
Studio — Claude 5-hour, crit at 12%. Fresh: plain number in its status color.
→
tick · +5s ▪La88·
Laptop — Codex at 88%, reading is aging. Trailing ·, dimmed grey.
→
tick · +10s ▪De63
Desktop — Codex at 63%, fresh. Then it wraps back to Studio at +15s.
Why a marker, not per-cell color, in side-by-side. A SwiftBar menu-bar line carries one color for the whole run — you can't tint each cell independently. So per-cell state rides the marker (which reads in monochrome anyway), and the line's single color tracks the binding machine — honest, since "how tight am I anywhere?" is what the color answers. Alternating sidesteps this entirely: one machine per tick means one cell, its own true color.

The five honesty states — legible even compact

This is the hard constraint. When the glyph shrinks, honesty can't. Each state keeps a distinct marker that reads in a monochrome bar, reinforced by color — the same vocabulary as the wide badge, miniaturized. Crucially, no-reading and offline carry no number — the badge never invents a zero for a machine it can't read.

A dash, muted, never a number: ▪ —. No limit reading captured yet. The badge won't fake a zero — it says so plainly.

A slashed circle, offline-grey, never a number: ▪ ⊘. A selected machine that goes offline is shown with this marker in the glyph it was selected for — marked, never silently dropped.

The never-a-number guard is structural, not cosmetic. The offline and no-reading cells have no code path that emits a digit — you can prove it by searching the emitted line for a number and finding none. This is the founding "be honest in the UI" rule, carried all the way down to the smallest glyph.

The tool mark — two neutral glyphs by default, real logos opt-in

The badge has always named the tool with a letter — C for Claude, X for codeX. This upgrades that default cue to two distinct neutral glyphs — a filled diamond for Claude, a filled triangle for Codex — chosen because they're different silhouettes (not one shape re-filled), solid (reads better than an outline at menu-bar size), and instantly told apart in a monochrome bar. They're used everywhere the tool is named: which tool binds a machine, and — new below — which tool an aggregate is. For those who want them, real product logos are an opt-in on top; the neutral glyph is always the xbar-safe floor underneath.

Why these shapes: the diamond and triangle share the geometric family of the ▪ mark but are unmistakably different outlines — the failure mode to avoid is two marks that differ only by fill (◇ vs ◆), which blur at menu-bar size. C / X remains a perfectly good fallback if you'd rather keep letters — this is a "clearly better or don't bother" call, flagged for you.
Logos are honest and opt-in. They render as small monochrome-friendly template images (SwiftBar image= / templateImage=) — SwiftBar-only polish. The neutral glyph is always the floor: on xbar, or if the image can't load, the diamond/triangle still names the tool. The Engineer supplies the assets; the stand-ins above show only the treatment, size (~13px), and placement (in the tool-cue slot, replacing the neutral glyph).

Group by tool — "all Claude" vs "all Codex" across your machines

A new Group choice flips what the glyph counts. Grouped by Host (today) each unit is a machine. Grouped by Tool each unit is a per-tool aggregate across the selected machines: all Claude = the tightest Claude window anywhere in the fleet; all Codex = the tightest Codex. The same layout and density you already picked apply — just to two tool-units instead of N machines. Same fleet as before: Studio (Claude 12%, binding), Laptop (Codex 88%), Desktop (Codex 63%) → all Claude = 12% (from Studio), all Codex = 63% (Desktop is tighter than Laptop's 88%… wait — 63 < 88, so Desktop binds Codex).

Tool · alternating — the bar rotates between the two aggregates every ~5s

Grouped by tool, alternating rotates between exactly two units — all Claude and all Codex — one per tick. Each frame is that aggregate's own compact (or wide) glyph, its own color, its own honesty state.

→

↻

The Hosts axis still scopes the aggregate. Grouping by tool doesn't discard the host selection — "all Claude" means the tightest Claude across the machines you've selected. Pick two build boxes and group by tool, and you get "Claude vs Codex across those two." Polling and the dropdown are unchanged — the dropdown still lists every machine with its full per-tool picture; only the glyph re-groups.
Aggregate honesty, precisely. An aggregate's number is the minimum remaining of that tool's windows across the selected hosts (the tightest), and its state is that tightest contributing window's state (fresh / aging / stale). A tool with no reading on any machine → — (never a zero). If every contributing host is offline, the aggregate reads offline ⊘. One tool's state never flags the other's — same per-unit independence as the per-host glyph.

The Display submenu — presets up front, axes underneath

A new 🖥 Display item joins the dropdown, riding the same shared action path as the host and service controls — so it appears in both single-host and multi-host modes. It opens onto a small set of named presets (the friendly front), then the three axes — Hosts, Layout, Density — each an enumerable choice. The active choice on each axis and the active preset are marked with ✓, read live from the config, so the menu never lies about your current setting.

How to read the submenu, top to bottom:
  • Presets come first — one click sets every layout axis and shows a tiny live preview of the glyph each produces. Six now: four grouped by host, two by tool. Here Compact icons side-by-side is active (✓), because the axes below exactly match its combination.
  • Hosts is a multi-select — each machine toggles in or out of the glyph. All hosts is the sentinel that clears the list. Here all three are individually checked (equivalent to "all", but pinned explicitly).
  • Layout and Density are single-choice — Side-by-side and Compact are marked. Change one and only that axis moves; the preset simply stops reading active if the trio no longer matches a named one.
  • Every choice writes the local config directly — no typed dialog, no HTTP — and the badge refreshes so the glyph and these ✓ marks update on the next render.
Selection is a view filter, never a monitoring change. Un-checking a machine here only removes it from the glyph. The poller still watches every machine, and the dropdown above still lists them all in full. A selected machine that goes offline is still shown in the glyph — with its ⊘ marker — never dropped from the view you chose.

Complete by design. The key names every symbol the badge can show — a legend that omits one is worse than none. Six freshness/offline states, the three status colors with their plain-word thresholds, what the number means, both tool marks, the side-by-side cues and overflow, and the Display submenu's ✓. One short line each — a scannable key, not paragraphs.
Why a submenu, not an expand-in-place. A toggle-expand would need the plugin to remember it's open (a legendOpen marker written and read each render) — more state, more to round-trip, and it re-renders the whole dropdown. A submenu is SwiftBar's built-in reveal: zero state, instant, and it keeps the default dropdown clean. It also works identically on xbar.
Placement. The Legend row sits in the shared action cluster, next to ☰ Watching and above ⊘ Uninstall — the "info / manage" band of the menu, below the live readings and the Display controls. Present in both modes via the same shared path.

Compact glyph grammar — the binding spec

What the Engineer implements. Every marker + color reuses the shipped honesty vocabulary, miniaturized. The marker is the load-bearing carrier (reads in monochrome); color reinforces. The ▪ leads the whole glyph once. Colors are the design-system status tokens lifted for the dark bar (good #5bd88a, warn #f0a94b, crit #ff6b6b).
